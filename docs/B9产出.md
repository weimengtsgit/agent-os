# AI Agent 底座（Agent OS）

## 30 / 60 / 90 天执行计划（评审版）

---

## 一、总体结论（一句话）

> **建议以 agno 作为“Agent Runtime 实现之一”，构建一个 Spec-first、可解耦、可治理的 Agent OS 底座，30 天跑通样板，60 天平台化，90 天可生产化与可迁移。**

---

## 二、总体目标与原则

### 🎯 总体目标

* 支撑 **多个业务领域 Agent 并行开发**
* 平台与业务 **职责清晰、互不拖累**
* **不被单一 Runtime（agno）锁死**
* 达到 **企业级生产可用**（稳定 / 安全 / 可审计）

### 🧭 核心原则

1. **Spec-first**：契约先行，代码跟随
2. **Runtime 可替换**：agno ≠ 平台
3. **平台最小化**：只做“必须统一的事”
4. **人始终在关键决策点**（Human-in-the-loop）

---

## 三、30 天计划（Phase 1：跑通 + 证明价值）

> 🎯 目标：**证明这不是 PPT 架构，而是可工作的 Agent 底座**

### 1️⃣ 核心交付物（30 天）

✅ **样板 Agent 上线（测试用例生成）**

* 输入：需求 ID / 文本
* 输出：结构化测试用例（JSON/Markdown）
* 包含：生成 → 校验 → 人审 → 回写

✅ **Agent OS MVP 能力**

* AgentSpec / ToolSpec / RunEvent / PolicySpec（JSON Schema）
* PAL（Platform Abstraction Layer）
* agno Runtime Adapter
* events.jsonl 事件流（可回放）

✅ **最小 Console**

* Agent 列表
* Run 列表
* Run 事件流
* 人审 approve / reject
* 成本 & 耗时汇总

---

### 2️⃣ 组织分工（30 天）

**平台组**

* 架构、Spec、PAL、Runtime Adapter
* Registry、事件流、CLI、Console
* CI / 门禁

**业务组（测试域）**

* 测试用例生成 Agent
* Prompt / 规则 / 质量判断
* 人审标准

---

### 3️⃣ 30 天验收标准（评审用）

* [ ] 不写一行业务代码即可新增 Agent（Spec + Prompt）
* [ ] Agent 全过程可回放（events.jsonl）
* [ ] 人审可控，不是黑盒自动化
* [ ] 不直接依赖 agno 的业务代码
* [ ] Demo 可在 5–8 分钟内讲清楚

---

## 四、60 天计划（Phase 2：平台化 + 多 Agent）

> 🎯 目标：**让多个业务组能“安全地一起用”**

### 1️⃣ 核心交付物（60 天）

✅ **平台化能力**

* Agent Registry / Catalog
* Agent 生命周期（draft → dev → canary → prod）
* Prompt / Model / Tool 动态配置
* Session / Run / State 持久化
* Policy（工具权限 / 限额）

✅ **多 Agent 并存**

* 至少 **2–3 个不同领域 Agent**

  * 测试用例生成
  * 测试计划生成 / 缺陷分析（任选）
* 并行运行互不影响

✅ **平台级 Observability**

* 成功率 / 失败率
* P95 耗时
* Token / Cost（Agent / Team 维度）

---

### 2️⃣ 组织协作模式（60 天）

**平台组**

* 只负责：Spec / Runtime / 治理 / 稳定性
* 不介入业务 Prompt 细节

**业务组**

* 自主开发 Agent
* 提交 Spec + Prompt + Tool
* 对输出质量负责

**关键变化**

> 平台组不再是“审批瓶颈”，而是“规则制定者 + 兜底者”

---

### 3️⃣ 60 天验收标准

* [ ] 新业务 Agent ≤ 1 天接入
* [ ] Agent 版本可回滚
* [ ] Agent 失败可定位（事件 + trace）
* [ ] 成本可按 Agent / Team 统计
* [ ] 无“绕平台直接用 agno”的情况

---

## 五、90 天计划（Phase 3：生产化 + 可迁移）

> 🎯 目标：**敢上生产、敢长期用、敢替换 Runtime**

### 1️⃣ 核心交付物（90 天）

✅ **生产级稳定性**

* 并发 / 限流 / 取消 / 熔断
* 工具雪崩保护
* Run 级超时

✅ **安全与合规**

* Secrets 管理（不进 Spec）
* 审计日志（Agent / Tool / 人审）
* 数据脱敏 / 输出过滤

✅ **成本与 SLO**

* Agent 成本预算
* 成本异常告警
* SLO：成功率 / 延迟 / 成本

✅ **Runtime 可迁移能力**

* Runtime Router
* Shadow Run（双跑）
* GameDay 演练（agno 异常 → 切换）

---

### 2️⃣ 90 天验收标准（“是否可长期投入”）

* [ ] 单 Agent 异常不影响其它 Agent
* [ ] 成本可控，有上限
* [ ] 审计满足合规要求
* [ ] Runtime 可切换（至少演练一次）
* [ ] 平台能支撑 5+ Agent 并存

---

## 六、人员与角色建议（最小配置）

| 角色           | 人数   | 职责                            |
| ------------ | ---- | ----------------------------- |
| 平台 Tech Lead | 1    | 架构 / 方向 / 决策                  |
| 平台工程师        | 2–3  | PAL / Runtime / Registry / CI |
| 业务 Agent 负责人 | 每域 1 | Agent 设计 / Prompt / 质量        |
| QA / 安全      | 兼职   | 评审 / 合规                       |

> ⚠️ **反模式**：让业务组自己维护 Runtime / 平台

---

## 七、主要风险与对策（评审关注点）

### 风险 1：平台过重、交付慢

👉 对策：30 天只做样板 + MVP，严禁“全能力一次性”

### 风险 2：被 agno 锁死

👉 对策：Spec-first + PAL + Contract Tests（已设计）

### 风险 3：成本不可控

👉 对策：90 天前必须有成本统计 + 限额

### 风险 4：平台组成为瓶颈

👉 对策：自动门禁（CI / Spec / Eval），减少人工审批

---

## 八、最终评审建议（给领导的）

**建议批准：**

* 以 **3 个月** 为一期投入
* 以 **测试用例生成 Agent** 为样板
* 以 **agno 作为 Runtime 实现之一**
* 以 **Spec-first Agent OS** 为长期方向

**3 个月后可决策：**

* 是否扩大 Agent 覆盖面
* 是否引入第二 Runtime
* 是否对外部团队开放平台
